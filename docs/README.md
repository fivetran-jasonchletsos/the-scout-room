# Scout Room — Operator Docs

This folder is the operator-facing kit for the Scout Room demo. It exists so
anyone touching the demo can answer the same questions confidently: what
audience are we serving, what story are we telling, where does each number
come from, and what should the next ODI demo we build look like.

The demo itself is a Fivetran Open Data Infrastructure (ODI) showcase: 150
years of MLB data flowing through Iceberg on S3, dbt on Snowflake, and
Cortex SEARCH / COMPLETE / ANALYST. A fan enters their hometown and the
Snowflake side stitches together a personalized scout report — no two are
the same. The architecture is fixed; the docs in this folder explain that
architecture, the demo around it, and how to reuse the structure.

## What's in this folder

- **`one-pager.md`** — single-page summary. Hand this to an AE, a prospect,
  or a leader who has three minutes. Covers the hook, what ODI does, the
  audience cuts, the tech stack, what the report actually outputs, and the
  business case for ODI customers.

- **`demo-runbook.md`** — operator script. Two timing tracks (5 minutes
  and 15 minutes), recommended hometown inputs that produce the most
  compelling reports, FAQ for the moments when a prospect veers off
  script, plus failure-mode guidance.

- **`data-lineage.md`** — for the technical questioner. Maps every panel
  in the Scout Report to its dbt model, its Cortex artifact, and the
  source columns it ultimately reads from. Use this when a data engineer
  on the prospect side asks where a specific number comes from.

- **`demo-template.md`** — the propagation kit. The Scout Room is the
  prototype for a family of ODI demos. This file abstracts the structure
  so the next demo (sales pipeline, support deflection, supply-chain
  signals, whatever) can be built on the same spine without rediscovering
  it from scratch.

## Reading order

- New to the demo? `one-pager.md` first, then `demo-runbook.md`.
- Selling the demo? `demo-runbook.md` and `data-lineage.md` open in tabs.
- Building the next demo? `demo-template.md`, then revisit `data-lineage.md`
  to see how the Scout Room maps the abstract pattern.

## Customer-facing artifacts

These live outside `docs/` because they ship as part of the app:

- **`/architecture`** — the visual architecture page in the running app.
  Hero pipeline diagram with animated particles, the Iceberg core + side
  view, the Cortex hub-and-spoke. This is the screenshot-worthy slide and
  the primary "wow" surface. Use it in pitches; the static images become
  slide bodies.
- **`/ref-doc`** — single-page printable summary in the brand. Linked from
  the landing pages alongside "watch / launch demo / git repo / Ref Doc".
- **`public/scout-room-ref-doc.pdf`** — pre-rendered one-page PDF of
  `/ref-doc`, generated by `scripts/build_pdf.sh` via headless Chrome
  (no npm deps). Stable URL for landing pages: `/scout-room-ref-doc.pdf`.
  Re-run the script whenever the ref doc content changes.

## What this folder does not do

These docs describe the demo, not the ODI architecture itself. The
Fivetran → Iceberg → dbt → Cortex flow is the product story — we
document it here, we do not redesign it. Architectural improvements
inside the demo are always additive (new marts, new UDFs, new UI panels,
new sources), never substitutive.
